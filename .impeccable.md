# Impeccable Design Context — CT Translate Extension

## Design Context

### Users
- **Primary user**: a professional reader working with foreign-language articles
  (English / Japanese / others) and translating them into **Traditional Chinese**.
- **Context of use**: daytime reading and research on long-form articles and news pages.
  The extension lives *on top of* the page being read — it must never compete with the
  article for attention.
- **Jobs to be done**:
  1. Hover + hotkey → translate a single paragraph in place.
  2. Translate a whole page, rendered inline beneath each paragraph (bilingual).
  3. Pick translation provider → then model, from a popup.
  4. Pick a translation *style* (a pre-authored prompt) from a popup dropdown.
  5. Author / manage those style prompts on a dedicated settings page.
- **No accounts, no login.** All config + API keys live in `chrome.storage.local`.

### Brand Personality
- Three words: **precise · quiet · instrument-grade**.
- It should feel like a well-machined tool — confident, dense with control where the power
  user wants it, but it gets out of the way during reading. Calm confidence, not flashy.
- Emotional goal: trust and control. The user is a professional handling source text; the
  UI should feel exact and dependable, never toy-like or "magical."

### Aesthetic Direction
- **Direction**: precise / instrument-grade structure, warmed by a **macaron** palette, in
  **light theme** (the harder, more distinctive path — most "technical" UIs reflex to
  dark+neon; we deliberately don't). The tension is the point: exact controls + soft color.
- **Theme**: Light only for v1. (User selected 亮色.)
- **Anti-references** (explicitly avoid):
  - Generic AI look: cyan-on-dark, purple→blue gradients, neon accents, glassmorphism,
    endless identical icon-over-heading cards.
  - Anything that disturbs reading: the on-page translation UI and hover affordance must be
    extremely restrained.
- **Reference feel**: as mature/legible as Immersive Translate, but **more refined than
  read-frog** (which leans utilitarian/developer-ish).
- **Differentiator / the one memorable thing**: **macaron as a functional color system.**
  A soft macaron palette is not decoration — each translation provider (OpenAI / Claude /
  Gemini / Google) and each saved translation style carries its own macaron hue as an
  identity chip. Color *encodes information*, which is what keeps a pastel palette feeling
  precise/instrument-grade instead of generic-soft. The pastels live on chips, tags, and
  surfaces; never on body text.

### Typography (direction — confirm at build time)
- Reflex fonts rejected (Inter, IBM Plex *, Space Grotesk/Mono, DM *, Fraunces, etc.).
- **Display / headings (Latin)**: `Archivo` — geometric grotesque, technical precision,
  variable weights for dense labels.
- **Body / UI (Latin)**: `Hanken Grotesk` — humanist grotesque, highly legible at the small
  sizes a control panel needs.
- **Mono (accents only)**: `Spline Sans Mono` — reserved strictly for model IDs, token
  counts, and the API-key field. NOT used as blanket "technical" decoration.
- **CJK (Traditional Chinese UI)**: `Noto Sans TC`. On-page injected translation text
  inherits the host page's font so it blends into the article.

### Color (direction — light, OKLCH, macaron functional palette)
- **Foundation (the 60%)**: warm cream "paper" background (not #fff) + warm cocoa/charcoal
  "ink" text (not #000). The neutrals carry warmth so the pastels feel native, not pasted on.
- **Macaron set** — soft pastels at low chroma (~0.05–0.09) and high lightness (~88–92% L).
  Each has a deeper *companion* (same hue, L ~55–62%, higher chroma) used ONLY for
  interactive emphasis, focus rings, and text-on-pastel. Working set:
  - Rose 玫瑰 · Pistachio 開心果綠 · Lavender 薰衣草紫 · Lemon 檸檬黃 · Blueberry 藍莓 · Peach 蜜桃
- **Functional assignment** (color = data, not mood):
  - Providers/styles map to stable macaron hues → used as chips/tags for instant identity.
  - "Active / now / here" state uses the active item's own companion color (not one global
    accent) — e.g. the selected provider's deeper hue drives its toggle + focus ring.
- **Discipline (so it never reads as AI-pastel slop)**:
  - Pastels on chips/tags/surfaces only — NEVER on body text (use ink).
  - One dominant neutral; pastels are the 10–30%, not a rainbow wash. No more than 2 macaron
    hues competing in a single view besides the neutral.
  - Keep the precise structure: tight grids, exact controls, sharp small-radius corners on
    dense controls. Pastel ≠ bubbly. Don't round everything; don't blur everything.
  - NO gradient text, NO pastel-on-pastel gradients, NO side-stripe accent borders,
    NO glassmorphism. Contrast for text always meets WCAG AA via ink-on-paper.

### Surfaces to design
1. **Popup** — provider dropdown → model dropdown, style dropdown, target language,
   translate-page action, node-translate toggle. Dense but calm.
2. **Options / settings page** — provider keys + models; **style prompt manager** (author,
   name, edit pre-set prompt templates with token placeholders).
3. **In-page injected UI** — hover affordance + inline bilingual translation rendering.
   Maximum restraint; Shadow DOM isolated.

### Design Principles
1. **Reading is sacred** — on-page UI defers to the article; chrome is for the popup/options.
2. **Precision over decoration** — every control is exact; no ornament that isn't a signal.
3. **Color is information** — macaron hues identify providers/styles and signal active state;
   they are never mood decoration. Pastels on chips/surfaces, ink on text.
4. **Dense where it's powerful, calm where it's read** — control panels can be information-rich;
   reading surfaces stay quiet.
5. **Never look AI-made** — pass the AI-slop test; warm cream + ink + disciplined functional
   macaron, not dark + neon and not a soft pastel wash.
